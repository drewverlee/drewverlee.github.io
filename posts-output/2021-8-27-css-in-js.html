CSS: CSS-in-JS

more on cljss in js

There are a multiple conclusions here, but a major one that seems to be forming is that for clojure devs, on teams without dedicated designers providing css classes, your fine to just use inline styles. That means your styles can be just clojure. Which is a great programming languge for organizing data and nothing on the market: LESS, SASS, CSS pseduo-elements, XYZ TOOL that comes out in 2040 will be better then it. Let's consider a counter argument from [here]:

It also means I can see little to no reason I can see to use tools like TailWind. Which provide deadcode elimination and minification, that we already get from Google Closure and data grouping (like hashmaps) that we already get from clojure. It also doesn't function as a styleguide. It obscures meaning for the sake of saving keystrokes and limits options. I think it and it's kin are a solution to a mix of problems, and as we know the path to enlightenment is to pull problems apart, not lump them together.

So fear not, you can use inline styles with confidence! Unless i'm wrong, feel free to comment.

Regardless, if your on the hunt for a way to have your cake and eat it to then you should consider any number of clojure libraries that will let you write css in js/cljs and create stylesheets.

  • Cljss Which compiles the styles from clojure structures to css classes and a style sheet. It has the downside that further composability has to happen through string manipulation.
  • stylefy. Which allows for composing styles via clojure. It seems to do load with inline styles and then creates and inserts classes and force another dom update?

I also hear lambda Island is creating a similar library. However, for my part, I'm still confused on how to justify the cost of any of them. What measurable problem is being solved and is it worth introducing another layer of abstraction? They seem to answer the question, can we? But I haven't seen a sufficient analysis of should we.
